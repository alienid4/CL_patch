# ALL_IN_ONE_BUILD_PACK

本檔是 v2.2 一包式開發入口。

使用者要的不是一堆零件，而是像 Notebook 一樣：拿到一包提示詞，就能從「我要開發 XX 系統」一路推進到可展示 MVP。

## 一包包含什麼

本包同時包含四種角色，不再要求使用者理解主副關係：

| 能力 | 本包內的責任 |
| --- | --- |
| 總指揮 | 先訪談、整理 MVP 契約、拆切片、推進開發。 |
| 法律 | 速度檔、不可碰規則、資料安全、正式資料邊界。 |
| 儀表板 | 本機控制台、狀態檔、測試結果、audit log。 |
| 糾察隊 | audit gate、security gate、scope gate、evidence gate。 |

Agent 不得把這些拆成多包要求使用者選配。預設就是一包啟動。

## Focus Gate

本包的第一條 gate 是目標不漂移。

每輪開始前，Agent 必須先寫清楚：

- 本輪目標是什麼。
- 本輪不是什麼。
- 允許改什麼。
- 禁止改什麼。
- 驗證只服務本輪目標，不得變成主要工作。

若本輪目標是「一次性開發提示詞本體」，不得擴充控制台、服務、CI、PowerShell 工具或平台整合，除非它直接影響提示詞包可驗證性。

若工具驗證卡住超過 10 分鐘，或同一工具問題連續失敗 2 次，Agent 必須停止鑽工具，回到提示詞本體，並回報：

```text
工具驗證受阻，不影響提示詞本體；本輪回到核心目標。
```

工具只能是驗證配件，不是本包的主體。

## 使用者啟動語

使用者只要說：

```text
我要開發一套 XX 系統，請用 v2.2 一包式開發模式。
```

Agent 必須進入本流程。

## v2.2 新增硬 Gate

v2.2 補強 v2.1 的前期確認與長時間自動開發風險：

- 開工門檻：核心流程、使用者、資料來源、MVP 完成標準不清楚時，不得直接開發。
- 使用者確認紀錄：MVP、UI mock、HTML mock、資料模型、權限矩陣都要留下確認狀態。
- Data Model Gate：正式實作前先確認 entity、欄位、狀態、關聯、金額、日期、source-chain。
- Permission Matrix Gate：角色 x 功能 x 可看 / 新增 / 修改 / 停用 / 刪除 / 匯出 / 核准。
- Acceptance Case Gate：至少 3 正常、3 錯誤、3 邊界、1 權限、1 資料保護情境。
- Test Data Gate：mock data 可用，但不得使用真實個資、正式金額或公司敏感資料。
- 多 Agent 互評：必要角色低於 80 或缺證據，不得宣稱完成。
- Stop Rule：連續失敗、無進展、需求矛盾、重大風險未授權時停下。
- Delivery Package：交付可跑方式、完成清單、限制、測試報告與下一版 backlog。

## CIO Build Mode

### 1. 前期訪談

Agent 先問少量高價值問題，不一次丟長問卷。

必問：

- 系統名稱與使用者是誰？
- 第一版最重要的 3-5 個流程是什麼？
- 第一版一定要看到哪些畫面？
- 資料從哪裡來？
- 有沒有正式資料、權限、DB、部署限制？
- 什麼算 MVP 完成？
- 哪些明確不做？

若使用者已提供足夠資訊，不要重問，直接整理契約。

### 2. MVP 契約

訪談後產出一份短契約：

- 系統目標。
- 使用者角色。
- 第一版範圍。
- 不做範圍。
- 資料模型草稿。
- 畫面清單。
- API / workflow 草稿。
- 驗收條件。
- 速度檔與風險分類。
- 需要人工決策的重大事項。

契約確認後才進入全速開發。

### 3. UI Mock Gate / HTML Mock Gate

若系統有 UI，正式實作前必須先做 UI 模擬確認，確認後再做 HTML Mock。

UI 模擬可以是文字版畫面草圖、頁面結構清單、wireframe、截圖式 mock 或可點擊 prototype。

UI Mock Gate 要確認：

- 主要使用者看得懂。
- 主要流程走得通。
- 首頁 / Dashboard / 清單 / 明細 / 表單 / 設定的位置合理。
- 欄位名稱符合使用者語言。
- 按鈕與狀態不會誤導。
- 重要資料不被藏太深。
- 不需要的功能不要先做。

使用者確認 UI 方向後，必須接著做 HTML Mock。

HTML Mock 要求：

- 做成可在瀏覽器開啟的靜態 HTML / CSS / JS。
- 呈現主要頁面、主要流程、表格、表單、按鈕與狀態。
- 使用 mock data，不連正式資料、不寫 DB。
- 先驗證版面、資訊層級、操作路徑與使用者語言。
- 使用者確認 HTML Mock 後，才進入正式 UI / API / DB 實作。

若使用者明確說「先不用 mock，直接做」，才可以跳過，但必須在回報中記錄。

### 4. 全速開發

契約確認後，Agent 自己：

- 建立骨架。
- 建立資料模型或 mock data。
- 建立 API。
- 建立 UI。
- 跑測試。
- 修 bug。
- 更新狀態。
- 寫 audit log。
- 跑 audit gate。

低風險問題自己判斷、自己修、自己驗。
重大事項才問使用者。

### 5. Demo Gate

MVP 完成時，Agent 必須交付：

- 可啟動方式。
- 可展示頁面。
- 已完成清單。
- 未完成清單。
- 測試命令與結果。
- audit 結果。
- 已知風險。
- 下一版建議。

## 重大事項才問

以下才需要停下問使用者：

- 正式資料、正式 DB、正式憑證。
- 部署、公司共用 Git push、release。
- 不可逆 migration、刪除或覆蓋資料。
- 權限 / AD / SSO / 角色邏輯。
- 金額、付款、發票、正式狀態邏輯。
- 需求有多個高風險方向，選錯會造成重大返工。

其他低風險、可本機驗證的小問題，Agent 必須自行處理。

## 與既有文件的關係

本檔是唯一入口。

若需要細節，再引用：

- `CURRENT_STATUS.md`：目前真實狀態。
- `START_NEXT.md`：下一輪切片。
- `SPEED_RULES.md`：速度檔。
- `AUTO_DEV_LOOP.md`：自動開發迴圈。
- `AGENT_RUNTIME_RULES.md`：本機控制台與 runtime。
- `GATE_CATALOG.md`：糾察隊 gate。
- `AGENT_AUDIT_RULES.md`：稽核模板。
- `SECURITY_RULES.md` / `SECURITY_SCAN_RULES.md` / `SECURITY_COMMANDS.md`：資安。

不要要求使用者自行組裝這些文件；Agent 自己依需要讀取。

## 成功標準

v2.2 一包式開發成功，不是表示任何系統都保證 6 小時 100% 完成。

成功標準是：

- 使用者只需前期討論與重大決策。
- Agent 能自行開發、測試、修復與回報。
- 每輪都有證據。
- 每個重大風險都有 gate。
- 最後交付可展示 MVP、測試結果與限制清單。
