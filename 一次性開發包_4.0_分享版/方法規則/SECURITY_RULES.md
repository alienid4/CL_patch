# SECURITY_RULES

本檔定義 AI_FEE 的日常資安、資料安全與 Agent 行為底線。

這不是漏洞掃描清單；漏洞掃描請看 `SECURITY_SCAN_RULES.md`。

## 核心底線

- 不碰正式資料、正式 DB、正式憑證，除非使用者明確授權且已有備份、rollback 與驗證方式。
- 不要求使用者提供正式密碼、token、private key、cookie、session、憑證或完整敏感資料。
- 不把 AI 判斷直接寫入正式金額、正式狀態或正式核准結果。
- 匯入 / 回匯必須先暫存、檢核、人工確認，再寫入正式表。
- archive 只能作為歷史參考，不得作為主程式來源。

## 不可提交或寫入版本控管

不得 commit 或寫入 Git：

- `.env`
- 密碼、token、API key、private key
- cookie、session、憑證
- 真實 DB 檔
- 真實 Excel / PDF / 發票 / 合約 / 簽呈原始檔
- 含敏感資料的 log
- 使用者個資或完整內部清單

若需要測試資料，使用假資料、遮罩資料、摘要資料或最小樣本。

## 正式資料規則

- 正式資料不得直接貼入對話或文件。
- 外部 AI 使用資料時必須去識別化。
- 可用數量、欄位名稱、遮罩值、hash、假資料描述問題。
- 若必須分析真實資料，需先取得使用者明確授權，並定義保存、刪除與回報方式。

## 權限與 DB 規則

- 第一版可用 SQLite 開發，但正式方向預設 Microsoft SQL Server。
- 不使用 SA / sysadmin 作為日常應用帳號。
- 正式 AD / SSO 未完成前，不得假裝已完成正式權限控管。
- 未來權限必須以後端 API 檢查為準，不能只靠前端隱藏按鈕。

## 高風險操作

遇到以下情況必須停下回報風險與需要確認事項：

- 正式部署
- 不可逆 migration
- 刪除大量資料
- 覆蓋既有資料
- 連線正式 DB
- 使用正式憑證
- 改登入 / 權限 / AD / SSO
- 匯入回匯可能覆蓋資料

## 回報要求

每個切片完成時，如有資安相關變更，需回報：

- 是否碰正式資料：是 / 否
- 是否新增敏感資料風險：是 / 否
- 是否需資安掃描：是 / 否，原因
- 是否需人工確認：是 / 否，原因
